CONTEXT

ClawSentinel's detection rules are the core of the security product. As of
beta14, 51 of 68 rules have no unit tests. Rules without tests can:

  - Contain silent false positives (flagging legitimate code)
  - Contain silent false negatives (missing the attack they claim to catch)
  - Regress undetected when other rules are modified
  - Use invalid samples that test the wrong rule or no rule at all

DECISION

Every new detection rule added to skill-scanner.ts REQUIRES a corresponding
Vitest unit test before the rule may be merged to main.

The test must:
  1. Include at least 2 TRIGGER samples (code that SHOULD be flagged)
  2. Include at least 2 SAFE samples (code that should NOT be flagged)
  3. Validate all samples via scanSource(), not raw regex
  4. Pass the test runner before the PR is approved

ENFORCEMENT

  - CI gate: vitest.config.ts enforces 80% branch coverage on skill-scanner.ts
  - PR requirement: any PR adding a rule without a test is rejected
  - Retroactive: OPEN-005 tracks completion of tests for existing 51 untested rules

RATIONALE

A rule that catches no real attack while flagging legitimate code creates
user friction without security benefit. A rule that misses attacks creates
false confidence. Tests are the only reliable way to verify both conditions.

CONSEQUENCES

  - Every rule addition now requires ~30 minutes additional work for tests
  - OPEN-005 must be completed before v1.0.0 (M4 blocker)
  - Grok is assigned OPEN-005 completion

POST-HOC REVIEW REQUIRED

Grok must review ADR-007 for testability constraints in beta14:
  - Are there rule types where scanSource() is insufficient for validation?
  - Are there edge cases the test structure should handle differently?
  - Does the 2+2 minimum sample count create gaps?
